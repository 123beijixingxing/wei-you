# 微友前端页面与接口清单

## 1. 文档目标

本文档基于 `docs/需求设计文档/微友开发需求文档.md` 继续拆解，面向产品、前端、后端、测试和项目管理使用，重点明确以下内容：

- `uni-app` 客户端的页面清单、路由规划、页面职责和关键交互。
- `Java` 后端的接口清单、接口边界、调用方式、鉴权要求和关键参数。
- 前后端联调时的公共约定，包括统一返回体、分页、上传、长连接事件和安全头。

说明：

- 当前仓库已有页面骨架包括 `pages/auth/login`、`pages/chat/index`、`pages/contacts/index`、`pages/discover/index`、`pages/me/index`、`pages/conversation/detail`、`pages/moments/index`、`pages/wallet/index`、`pages/feature-hub/index`。
- 本文档给出的是目标态页面和接口清单，允许与现有代码骨架分阶段对齐。

## 2. 前端页面拆分原则

### 2.1 分包建议

| 分包/模块 | 路径前缀 | 说明 | 优先级 |
| --- | --- | --- | --- |
| 主包 | `pages/auth`、`pages/chat`、`pages/contacts`、`pages/discover`、`pages/me`、`pages/conversation` | 登录和四大主导航、高频聊天页 | P0 |
| 社交内容包 | `pages/moments`、`pages/group`、`pages/profile` | 朋友圈、群资料、个人资料 | P0 |
| 钱包支付包 | `pages/wallet` | 零钱、账单、转账、红包、支付码 | P1 |
| 生态内容包 | `pages/official`、`pages/miniapp`、`pages/channels`、`pages/live` | 公众号、小程序、视频号、直播 | P2 |
| 工具能力包 | `pages/search`、`pages/scan`、`pages/location`、`pages/webview` | 搜索、扫码、位置、H5 容器 | P1 |
| 设置中心包 | `pages/settings`、`pages/feedback` | 隐私、安全、通知、帮助反馈 | P0 |

### 2.2 页面状态约定

- `guest`：未登录可访问，如登录页、协议页。
- `auth`：登录后可访问，如会话、联系人、朋友圈。
- `verified`：需要实名或安全校验后访问，如钱包、银行卡。
- `payment_auth`：需要支付密码、人脸或短信二次校验后访问，如转账、提现。

### 2.3 页面开发约定

- 页面级数据优先采用 `onLoad/onShow` 拉取首屏数据，长列表使用分页加载。
- 聊天、通知、通话等实时页面同时监听 `WebSocket` 事件和本地状态中心。
- 路由入参统一使用 `id`、`scene`、`from` 等短参数，复杂对象不直接通过 URL 传输。
- 页面内公共逻辑抽离到 `stores`、`composables`、`api` 模块，避免重复请求和状态散落。

## 4. 前后端公共接口约定

### 4.1 Base URL 与鉴权头

- API 前缀：`/api`
- 鉴权头：`Authorization: Bearer <token>`
- 设备头：`X-Device-Id`
- 渠道头：`X-Client-Type`，例如 `app`、`h5`、`mp`
- 追踪头：`X-Trace-Id`
- 幂等头：`X-Idempotency-Key`，用于支付、红包、转账、创建类接口

### 4.2 统一返回体

```json
{
  "code": 0,
  "message": "ok",
  "traceId": "20260509abc123",
  "data": {}
}
```

### 4.3 分页结构

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "list": [],
    "pageNo": 1,
    "pageSize": 20,
    "total": 120,
    "hasMore": true,
    "nextCursor": "1710000010000"
  }
}
```

### 4.4 文件上传约定

- 第一步：调用媒体策略接口获取上传凭证。
- 第二步：客户端直传对象存储。
- 第三步：将媒体元数据提交业务接口。
- 图片、视频、语音、文件均统一走媒体中心，避免业务模块重复上传实现。

### 4.5 WebSocket 约定

- 连接地址：`/ws/chat?token=xxx&deviceId=xxx`
- 心跳间隔：30 秒。
- 重连策略：指数退避，最多重试 8 次。
- 客户端消息格式：

```json
{
  "event": "MESSAGE_SEND",
```

## 6. WebSocket 事件清单

| 事件名 | 方向 | 说明 | 关键字段 |
| --- | --- | --- | --- |
| `CONNECT_ACK` | 服务端 -> 客户端 | 连接建立成功 | `userId`、`serverTime` |
| `HEARTBEAT` | 双向 | 保持长连接活跃 | `timestamp` |
| `MESSAGE_SEND` | 客户端 -> 服务端 | 发送消息 | `conversationId`、`clientMsgId`、`msgType`、`content` |
| `MESSAGE_ACK` | 服务端 -> 客户端 | 服务端确认收包 | `clientMsgId`、`messageId`、`sendStatus` |
| `MESSAGE_RECEIVE` | 服务端 -> 客户端 | 推送新消息 | `conversationId`、`message` |
| `MESSAGE_READ` | 双向 | 已读回执 | `conversationId`、`messageId`、`readerId` |
| `MESSAGE_RECALL` | 双向 | 撤回消息通知 | `messageId`、`operatorId` |
| `TYPING` | 双向 | 输入中状态 | `conversationId`、`status` |
| `CALL_INVITE` | 双向 | 音视频来电邀请 | `roomId`、`callType`、`memberIds` |
| `CALL_SIGNAL` | 双向 | WebRTC 信令交换 | `roomId`、`sdp`、`candidate` |
| `NOTICE_PUSH` | 服务端 -> 客户端 | 系统通知推送 | `noticeType`、`payload` |

## 8. 研发落地建议

- 前端先按本文档补齐 `pages.json` 与分包配置，再逐步落地页面骨架。
- 后端先实现 `P0` 接口，确保登录、聊天、联系人、朋友圈闭环可跑通。
- 接口响应字段在进入正式开发前应进一步固化为 Swagger/OpenAPI 文档。
- 长连接事件建议同步输出事件枚举、示例报文和异常码表，方便前后端调试。
