# 微友 App 开发需求文档

## 1. 文档说明

### 1.1 项目背景

`微友` 是一个基于 `uni-app` 前端和 `Java` 后端构建的高仿微信超级应用项目，目标是尽可能完整覆盖微信主端的核心能力、常用交互和生态功能，包括即时通讯、通讯录、朋友圈、发现、支付、公众号、小程序、视频号、直播、收藏、卡包、设置、安全中心等模块。

### 1.2 建设目标

- 构建与微信主端信息架构接近的移动端产品。
- 前端采用 `uni-app + Vue 3 + Pinia`，优先支持 Android 与 iOS App，同时保留 H5 与小程序兼容能力。
- 后端采用 `Java 21 + Spring Boot 3` 技术体系，支持高并发 IM、支付、内容流与生态扩展。
- 需求设计覆盖产品功能、页面结构、接口边界、核心数据模型、技术实现细节和非功能性要求。

### 1.3 需求范围

本需求文档按“目标态”覆盖微信主端公开可见的主要功能，并结合实际研发规律拆分为分阶段交付：

- 一期：账号体系、单聊群聊、通讯录、朋友圈、发现页基础能力、个人中心。
- 二期：语音视频通话、钱包支付、红包转账、扫一扫、搜一搜、公众号。
- 三期：小程序平台、视频号、直播、开放平台、后台运营、风控审核、灰度发布。

## 2. 产品定位

### 2.1 产品定位

`微友` 定位为“社交 + 通讯 + 内容 + 支付 + 生态服务”的超级应用，视觉和操作路径参考微信，但品牌、图标、文案、音效、素材需要使用自有设计资产，避免直接复用微信商标与版权内容。

### 2.2 目标用户

- 普通个人用户：聊天、社交、支付、内容浏览、便民服务。
- 商户与服务提供方：收款、公众号、小程序、视频号带货、会员经营。
- 内容创作者：朋友圈发布、视频号内容生产、直播互动。
- 平台运营人员：用户管理、内容审核、活动运营、风控处置。

### 2.3 核心指标

- 登录成功率 >= 99.9%。
- 文本消息端到端送达成功率 >= 99.99%。
- 消息实时到达平均延迟 <= 500ms。
- App 冷启动时间 <= 2.5s。
- 钱包交易核心链路可用性 >= 99.99%。

## 3. 技术架构设计

### 3.1 前端技术方案

- 框架：`uni-app + Vue 3 + Vite`
- 状态管理：`Pinia`
- 网络请求：统一 `request` 封装，支持 Token、签名、重试、幂等键透传
- 实时通信：`WebSocket`，语音视频信令采用 `WebSocket + WebRTC`
- UI 组织：主包 + 分包结构，主包承载四大 Tab，高频页面优先首包加载，钱包、直播、小程序、设置中心等放入分包
- 设备能力：摄像头、相册、地理位置、扫一扫、录音、文件选择、消息推送、系统分享

### 3.2 后端技术方案

- 核心框架：`Java 21 + Spring Boot 3 + Spring Cloud Alibaba`
- 网关鉴权：`Spring Cloud Gateway + Spring Security + JWT/OAuth2`
- 数据存储：`MySQL 8` 保存核心业务数据，`Redis` 负责缓存、会话、分布式锁、计数器
- 消息系统：`Kafka` 或 `RocketMQ` 支撑消息投递、异步任务、事件通知
- 搜索引擎：`Elasticsearch` 支撑搜一搜、聊天记录搜索、公众号/小程序检索
- 对象存储：`MinIO` 或云 OSS，存储图片、视频、语音、文件、头像、朋友圈媒体资源
- 实时服务：`Netty/WebSocket` 提供长连接，支持在线状态、消息投递、已读回执、撤回广播
- 音视频：`WebRTC + TURN/STUN`，后端负责房间管理、信令转发、通话鉴权
- 可观测性：`SkyWalking/Zipkin + Prometheus + Grafana + ELK`

### 3.3 架构演进建议

- 一期采用“模块化单体 + 独立 IM 网关”提高开发效率。
- 二期拆分为账号中心、关系链、IM 服务、内容服务、支付服务、生态服务、审核服务、搜索服务。
- 三期增加开放平台、运营平台、商户平台、推荐服务、A/B 实验与灰度发布系统。

## 4. 系统总体结构

### 4.1 客户端结构

底部主导航保持与微信近似的四大入口：

- 微信：会话列表、通知、置顶、草稿、未读数
- 通讯录：好友、群聊、标签、公众号、企业联系人、黑名单
- 发现：朋友圈、视频号、扫一扫、直播、附近、搜一搜、小程序、购物、游戏
- 我：个人主页、收藏、朋友圈、卡包、表情、设置、支付入口

### 4.2 后端模块划分

- `auth-center`：注册登录、设备管理、会话管理、账号安全
- `user-center`：用户资料、头像、状态、隐私设置、地区信息
- `relation-service`：好友关系、黑名单、标签、群成员关系
- `im-service`：会话、消息、消息序列、已读回执、消息撤回、草稿、置顶
- `group-service`：群信息、群公告、群权限、入群审批、群二维码
- `moment-service`：朋友圈发布、评论、点赞、可见范围、举报
- `wallet-service`：零钱、银行卡、账单、收付款、转账、红包、商户支付
- `content-service`：视频号、直播、短视频、点赞评论、推荐流
- `official-service`：公众号、文章、菜单、客服消息、模板消息
- `miniapp-service`：小程序应用、版本、路由、授权、支付打通
- `search-service`：综合搜索、分词、联想词、热搜词
- `audit-risk-service`：内容审核、设备风控、交易风控、账号风险识别
- `admin-console`：运营管理后台、审核后台、数据看板、配置中心

### 5.3 前端公共能力

- 全局会话未读角标管理

- 本地缓存最近会话、聊天草稿、最近搜索、表情与用户配置

- 支持深色模式预留，但默认优先交付浅色微信式体验

## 6. 核心功能需求

### 6.1 账号与登录体系

#### 功能点

- 手机号注册、手机号验证码登录、密码登录
- 微信式账号 ID、二维码名片、手机号换绑
- 找回密码、设备验证、短信验证、邮箱补充绑定
- 登录设备管理、最近登录记录、异常登录提醒
- 实名认证、支付实名、未成年人模式、账号冻结与解封

#### 实现细节

- 前端登录页支持手机号、密码、验证码三种切换方式，并保留协议勾选、国家区号、图形验证码扩展位。
- 后端生成 `accessToken + refreshToken` 双令牌，`Redis` 存储登录态和设备指纹。
- 设备管理需要记录设备型号、系统版本、登录 IP、城市、登录时间、推送通道标识。
- 异常登录采用设备指纹、IP 归属地、操作频率和风险分综合判定，可触发二次验证。
- 支付相关功能要求补齐实名、支付密码、人脸验证等高安全链路。

### 6.2 个人资料与社交名片

#### 功能点

- 头像、昵称、性别、地区、个性签名、状态、二维码名片
- 微友号修改、手机号展示控制、朋友圈权限设置
- 拍一拍文案、朋友圈封面、状态图标

#### 实现细节

- 用户资料分为公开资料、好友可见资料、仅自己可见资料三层。
- 头像、封面、状态图使用对象存储，上传后由媒体服务做压缩和敏感内容检测。
- 用户二维码名片支持长期有效码和动态短期码，动态码可配置有效期和一次性使用策略。

### 6.3 会话列表与消息中心

#### 功能点

- 单聊、群聊、服务通知、系统通知
- 会话置顶、免打扰、标为未读、删除会话、草稿提示
- 未读数角标、最后一条消息预览、@我提醒、撤回提示

#### 实现细节

- 会话列表以 `conversation_id` 维度聚合，前端进入页面先读取本地缓存，再调用接口增量同步。
- 后端维护用户会话表、未读数表、最后消息快照表，减少列表查询成本。
- 服务通知和系统通知采用单独会话类型，避免与普通社交会话混淆。
- 草稿保存在本地，并可同步到服务端以支持多端草稿续写。

### 6.4 单聊消息系统

#### 功能点

- 文本、图片、语音、视频、文件、名片、位置、链接、小程序卡片
- 消息转发、引用回复、撤回、删除、收藏、多选操作
- 已读未读、对方输入中、消息送达状态、失败重发
- 文件传输助手、自发消息、聊天背景、消息搜索

#### 实现细节

- 发送链路采用“客户端本地生成临时消息 ID -> 服务端返回正式消息 ID -> ACK 更新状态”模式。
- 实时通信采用 `WebSocket`，离线消息通过 `MQ + Push` 联动补偿。
- 消息表需区分 `msg_id`、`client_msg_id`、`conversation_id`、`seq_no`、`sender_id`、`msg_type`、`content_body`、`send_status`。
- 图片、视频、文件消息先上传媒体服务，再发送媒体元数据；语音消息需要保存时长、波形、转写文本扩展位。
- 撤回规则默认 2 分钟内可撤回，管理员和群主可配置特殊权限。
- 聊天记录搜索使用 `Elasticsearch` 建立索引，支持关键字、高亮、类型筛选。

### 6.5 群聊系统

#### 功能点

- 建群、邀请入群、扫码入群、群主转让、群公告、群昵称
- 群管理、禁言、移出群聊、入群审批、全员禁言、群二维码
- @成员、群接龙、群待办、群收款、群红包

#### 实现细节

- 群信息表与群成员表分离，成员关系单独维护角色、来源、入群时间、禁言状态。
- 群消息仍走统一 IM 服务，但消息投递目标改为群成员列表，在线用户实时推送，离线用户走补偿。
- 大群场景建议引入分片投递和消息写扩散控制，避免单群广播阻塞。
- 群权限模型至少包含群主、管理员、普通成员三级，可扩展企业群角色。
- 群二维码需要绑定群 ID、过期时间、扫码策略、入群验证方式。

### 6.6 语音与视频通话

#### 功能点

- 单人语音通话、单人视频通话
- 群语音、群视频会议、屏幕共享、麦克风与摄像头控制
- 来电提醒、通话中切后台、弱网提示、通话记录

#### 实现细节

- 使用 `WebRTC` 建立媒体通道，Java 服务负责信令、房间、成员状态和鉴权。
- 需要部署 `STUN/TURN` 服务解决 NAT 穿透；弱网场景下动态调整码率和分辨率。
- 通话邀请消息进入 IM 会话流，并在客户端转化为弹层或全屏来电页。
- 通话记录写入独立表，用于最近通话、账单归因和质量分析。

### 6.7 通讯录与关系链

#### 功能点

- 好友添加、通讯录导入、搜索添加、扫码添加、群内加好友
- 好友备注、标签分组、星标朋友、黑名单、删除好友
- 新的朋友、群聊列表、公众号列表、企业联系人入口

#### 实现细节

- 好友关系采用双向关系表，记录申请状态、来源、备注、星标、朋友圈权限。
- 标签体系支持用户自定义标签、好友批量归类、群发对象选择。
- 删除好友后默认保留聊天记录策略可配置，同时对方状态展示需按平台隐私规则处理。
- 通讯录导入涉及敏感权限，前端应给出明确授权说明，后端仅存储用户确认后的哈希匹配结果。

### 6.8 朋友圈

#### 功能点

- 发布文字、图片、视频、定位、提醒谁看、谁可以看
- 点赞、评论、删除、置顶自己的动态、删除评论
- 朋友圈封面、消息提醒、好友动态流、仅三天可见、分组可见

#### 实现细节

- 内容表拆分为主表和媒体表，文本、媒体、位置、可见范围分别结构化存储。
- 发布时先上传媒体资源，再生成动态；动态创建成功后异步写入粉丝时间线或采用拉模型聚合。
- 可见范围支持公开、私密、部分可见、不给谁看四种策略，后端查询流时实时过滤。
- 点赞评论写入互动表，并通过消息中心给动态作者和被 @ 用户推送提醒。
- 图片视频内容进入审核服务，支持涉政、涉黄、辱骂、广告、违规二维码识别。

### 6.9 发现页能力矩阵

#### 功能点

- 朋友圈入口
- 视频号入口
- 扫一扫
- 搜一搜
- 直播和附近
- 小程序最近使用
- 购物、游戏、城市服务等生态入口

#### 实现细节

- 发现页采用配置化卡片布局，后台可动态控制模块显示、排序、AB 试验和地区差异化投放。
- 各能力入口统一通过路由中心跳转，支持原生页、uni-app 页、小程序容器页和 H5 容器页。
- 入口点击数据需要埋点，用于推荐排序、热度分析和转化统计。

### 6.10 扫一扫

#### 功能点

- 扫二维码、条形码、付款码、收款码、群二维码、名片二维码
- 相册识码、扫码历史、扫码结果页

#### 实现细节

- 前端封装摄像头扫码组件，支持自动对焦、闪光灯、从相册导入、扫码动画。
- 扫码结果由后端统一解析二维码内容类型，包括用户、群聊、公众号、商品、支付、网页登录、小程序等。
- 登录二维码采用短期票据机制，PC 端展示二维码，App 扫码后需要二次确认登录。

### 6.11 搜一搜

#### 功能点

- 搜联系人、群聊、聊天记录、公众号、文章、小程序、视频号、朋友圈、商品、服务
- 热门搜索、搜索建议、历史搜索、分类搜索结果

#### 实现细节

- 后端以 `Elasticsearch` 建立多索引体系，不同业务域使用不同 analyzer 和权重策略。
- 搜索建议使用前缀匹配和热词模型，支持拼音、首字母、昵称别名联想。
- 历史搜索本地缓存与服务端同步并存，用户可清空历史。

### 6.12 公众号平台

#### 功能点

- 关注/取消关注公众号
- 公众号主页、历史消息、文章详情、菜单导航
- 自动回复、客服消息、模板通知、服务通知

#### 实现细节

- 公众号作为特殊关系对象管理，消息会进入单独服务会话。
- 文章内容支持富文本、长图、视频、外链和广告位配置。
- 菜单点击可触发消息回复、H5 跳转、小程序跳转、客服会话。
- 服务通知与公众号消息统一进入消息中心，但会按类型区分展示样式。

### 6.13 小程序平台

#### 功能点

- 小程序列表、最近使用、我的收藏、搜索发现
- 小程序授权登录、用户信息授权、支付授权、订阅消息
- 小程序分享卡片、聊天内打开、扫码打开、桌面快捷入口

#### 实现细节

- App 内需提供小程序容器层，支持路由、生命周期、JSBridge、权限隔离和资源沙箱。
- 小程序元数据包含应用 ID、版本号、页面路由、权限声明、审核状态、发布状态。
- 小程序支付、地理位置、摄像头、相册、录音等能力必须通过宿主权限网关统一代理。
- 需要建设开发者后台、提审发布后台、审核后台和违规下架机制。

### 6.14 视频号与直播

#### 功能点

- 短视频浏览、点赞、评论、转发、关注创作者
- 直播开播、直播列表、直播间弹幕、礼物、连麦、商品挂载
- 创作者主页、粉丝数据、内容管理、违规处理

#### 实现细节

- 视频流采用对象存储 + CDN + 转码服务，输出多码率 HLS/MP4。
- 推荐流可先按时间流 + 热度流实现，后续增加画像推荐和协同过滤。
- 直播需拆分推流、播放、聊天室、礼物、连麦、订单、审核等子模块。
- 弹幕和直播聊天建议单独频道处理，避免与普通 IM 会话互相影响。

### 6.15 钱包与支付

#### 功能点

- 零钱余额、银行卡管理、账单明细、支付密码
- 收付款码、二维码收款、转账到好友、转账到账提醒
- 红包、群红包、拼手气红包、退款、提现、充值
- 商户支付、订单支付、城市服务、生活缴费、发票与电子凭证

#### 实现细节

- 钱包账户、交易流水、冻结金额、可用余额分表设计，所有金额字段使用分为单位的整数存储。
- 交易核心链路使用强一致事务、幂等号、余额变更日志和对账任务保证一致性。
- 红包系统需要支持拆包算法、领取状态、过期退回、群红包并发抢夺控制。
- 收付款码采用动态 Token 和短时有效策略，避免被截屏长期复用。
- 绑卡、实名、人脸、支付密码修改、提现审核、异常交易拦截都要经过风控服务。

### 6.16 收藏、卡包与表情

#### 功能点

- 收藏文本、链接、图片、视频、聊天记录、文件、小程序
- 卡包管理：会员卡、优惠券、票券、交通码、门禁码
- 表情商店、最近使用、收藏表情、聊天大表情

#### 实现细节

- 收藏夹需要统一资源模型，资源原件可引用聊天或朋友圈媒体，也可独立存储副本。
- 卡包需要接入码制规则、核销状态、有效期、门店范围和商户系统对接。
- 表情资源需要 CDN 分发、包版本管理和下载缓存机制。

### 6.17 我的页面与设置中心

#### 功能点

- 个人主页、朋友圈入口、收藏、支付、卡包、表情、设置
- 通用设置、通知设置、隐私设置、聊天设置、青少年模式、帮助与反馈
- 多设备登录管理、存储空间清理、聊天记录迁移、消息免打扰

#### 实现细节

- 设置中心采用配置项驱动，方便后端灰度开启新能力。
- 聊天记录迁移支持本机备份、局域网迁移、云备份扩展位。
- 隐私设置至少支持加好友方式、朋友圈可见范围、黑名单、通讯录匹配开关。

### 6.18 文件传输助手与多端协同

#### 功能点

- 文件传输助手
- Web/PC 登录与扫码确认
- 多端消息同步、未读同步、已读同步

#### 实现细节

- 文件传输助手本质为用户和系统保留账号的单人会话，可用于图片、文件、链接、笔记中转。
- 多端登录时由会话中心统一管理设备在线状态，支持主设备下线其他设备。
- 多端同步以消息序列号增量拉取实现，避免全量刷库。

### 6.19 系统通知与推送

#### 功能点

- 好友申请通知、群通知、点赞评论通知、支付通知、系统公告
- App 推送通知、桌面通知、角标更新、铃声与震动

#### 实现细节

- 推送通道接入厂商推送和 APNs，按照设备类型动态选择。
- 用户处于前台时优先站内通知，后台或离线时触发系统推送。
- 推送内容遵守隐私策略，锁屏默认不显示敏感消息全文，可由用户自行配置。

### 6.20 后台管理系统

#### 功能点

- 用户管理、封禁解封、实名审核、设备风控
- 内容审核：朋友圈、视频、文章、直播、昵称头像
- 支付运营：交易查询、退款处理、对账、异常单排查
- 生态管理：公众号、小程序、视频号、商户审核
- 配置中心：发现页配置、活动配置、推送模板、敏感词库

#### 实现细节

- 后台建议采用前后端分离架构，通过 RBAC 权限模型控制菜单、按钮、数据范围。
- 审核系统支持机审 + 人审，记录审核版本、审核员、审核结论和申诉链路。
- 核心运营操作必须保留审计日志，涉及资金、封号、权限提升操作要求双人复核。

## 8. 核心接口设计原则

### 8.1 API 风格

- 业务接口以 `RESTful JSON` 为主。
- 实时聊天、在线状态、通话信令使用 `WebSocket`。
- 文件上传采用预签名上传或网关中转上传。
- 内部高频服务间调用可使用 `OpenFeign`，高并发路径可引入 `gRPC`。

### 8.2 接口安全

- 所有接口必须校验 Token、设备号、签名时间戳。
- 钱包、实名、支付、提现接口要求二次鉴权和风控令牌。
- 关键接口加入幂等键、防重放、防刷限流和黑名单机制。

### 8.3 典型接口清单

- `/api/auth/login/password`
- `/api/auth/login/sms`
- `/api/user/profile/get`
- `/api/contact/friend/apply`
- `/api/chat/conversation/list`
- `/api/chat/message/history`
- `/api/chat/message/send`
- `/api/moment/post/create`
- `/api/wallet/balance/get`
- `/api/wallet/transfer/create`
- `/api/search/global`
- `/api/official/account/detail`
- `/api/miniapp/open`

## 9. 非功能性要求

### 9.1 性能要求

- 会话列表接口 P95 响应时间 <= 200ms。
- 历史消息接口 P95 响应时间 <= 300ms。
- 普通图片上传成功率 >= 99.9%。
- 直播间聊天支持万级在线用户扩展。

### 9.2 安全要求

- 敏感数据传输使用 `HTTPS/TLS`，敏感字段脱敏展示。
- 登录密码、支付密码采用强哈希存储，卡号按合规要求加密。
- 具备设备指纹、行为分析、滑块、人机识别、短信频控等安全措施。

### 9.3 可用性要求

- 支持服务降级、限流、熔断、消息补偿、断线重连。
- 消息服务、支付服务、审核服务需具备多实例高可用部署能力。
- 媒体资源、配置数据、热数据需配置缓存与 CDN。

### 9.4 合规要求

- 支付功能需满足支付牌照、实名、反洗钱、对账留档要求。
- 内容平台需满足内容审核、未成年人保护、举报投诉、日志留存要求。
- 通讯录导入、位置、相册、摄像头、麦克风权限需明确弹窗告知并留存授权记录。

## 11. 风险与注意事项

- 高仿微信不等于直接复制微信素材，UI 结构可参考，但品牌资产、图标、文案、启动图必须原创。
- 全功能目标涉及 IM、支付、内容、直播、生态平台等多个高复杂度领域，必须按阶段推进，避免一次性铺开导致项目失控。
- 钱包、直播、开放平台、小程序、内容审核均涉及合规审批与第三方资质，应在立项初期同步准备。
- 多端一致性、消息可靠投递、红包高并发、直播弹幕、搜索索引和风控识别是重点技术难点，应优先预留架构能力。

## 12. 交付建议

- 先交付可运行的 `uni-app` 主框架和 `Spring Boot` 模块化后端骨架。
- 先打通登录、会话、联系人、朋友圈四条主链路，确保用户可完成注册、加好友、聊天、发动态的完整闭环。
- 支付、小程序、视频号、直播等高复杂度功能采用独立服务和独立项目计划管理。
- 所有模块统一埋点、日志、审计、灰度和配置中心，确保后期可持续迭代。
